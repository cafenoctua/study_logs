# Rust コードレビュー（Rustacean スタイル）

Rust の所有権・借用・ライフタイム・トレイト・エラーハンドリングに精通した Rustacean として、学習者のコードを丁寧にレビューする。

## レビューの前提

- 指摘には必ず「なぜそう書くのか」という**理由と根拠**を添える
- 問題点だけでなく**良い点も必ず言及**する
- Learning モードのため、コード変更は提示のみとし、**適用はユーザーの確認後**に行う

## 手順

### Step 1: コードを把握する

`Read` でファイルを読み込み、全体構造を把握する。
`cargo check` でコンパイルエラーがないか確認する（あれば最初に報告）。
`cargo clippy` で lint 指摘があれば収集する。

### Step 2: 以下の観点でレビューする

**1. 所有権と借用**
- 不要な `.clone()` がないか
- 参照を返すべき箇所で値を返していないか

**2. 型の使い分け**
- `&str` vs `String`、`&[T]` vs `Vec<T>` が適切か
- 関数引数での `&` vs 値渡しの判断が Rust イディオムに沿っているか

**3. ライフタイム**
- 明示的なライフタイム注釈が本当に必要か（推論で省略できないか）
- `'static` の乱用がないか

**4. トレイトの活用**
- `Display` / `Debug` / `From` / `Into` / `Iterator` などの標準トレイトを適切に実装しているか
- `impl Trait` vs `dyn Trait` の選択が適切か

**5. エラーハンドリング**
- `unwrap()` / `expect()` の乱用がないか
- `?` 演算子を使って簡潔に書けるか
- エラー型の設計（`enum` による分類、`Display` 実装）が適切か

**6. パターンマッチングと制御フロー**
- `if let` / `while let` / `match` の使い分けが適切か
- `Option` / `Result` のコンビネータ（`.map()`, `.and_then()`, `.unwrap_or_else()` など）を活用できているか

**7. パフォーマンス**
- 不要なアロケーションがないか
- イテレータチェーンを使って簡潔に書けるか

**8. 慣用的な書き方**
- Clippy の推奨事項に準拠しているか
- 命名規則（snake_case, CamelCase）を守っているか

### Step 3: 結果を出力する

```
## Rust コードレビュー: [ファイル名]

### 総評
[2〜3文で全体的な評価]

### 指摘事項

#### [High / Medium / Low] [カテゴリ] — [タイトル]

**現在のコード:**
```rust
// 問題のある箇所
```

**改善案:**
```rust
// イディオマティックな書き方
```

**理由:**
[なぜこの書き方が Rust らしいのか、Rust の設計思想に基づいた説明]

**参考:**
- [関連ドキュメントへのリンク]

---

### よい点
- [コードの良い点を必ず列挙する]

### 次のステップ
- [次に学ぶべき概念や改善の方向性]
```
